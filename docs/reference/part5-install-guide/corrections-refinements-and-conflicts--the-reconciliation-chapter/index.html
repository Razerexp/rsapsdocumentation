59. Corrections, Refinements, and Conflicts — The Reconciliation Chapter

Every point where the official guide changes something stated elsewhere in this field guide, each with a verdict. These are the entries to read before quoting any spec to a customer.

59.1 CORRECTION — Prime Server Sizing​

Section 9.2 (and the Section 17 appendix and Section 22 runbook) record the session-stated minimum of 8 vCPU / 8 GB RAM. The official guide specifies 8 vCPU / 16 GB RAM / 60+ GB disk (Red Hat 7.x/8.x or Windows, 64-bit, customer-provided OS), and its Windows server-resources table goes further: Windows 2019 or later, 100 GB disk, 16 GB, 8 CPUs. Verdict: use the official numbers — 16 GB is the floor. The transcript's 8 GB likely reflects an older prerequisite document; notably, the session's own field anecdote (a customer growing 8 → 16 → 30 GB under load) points the same direction. Sections 9.2 and 17 now carry inline pointers to this correction; the readiness checklist (19.1) and runbook (22.1) should be read with 16 GB / 60–100 GB substituted.

59.2 REFINEMENT — Load Balancer Persistence: Three Sources Reconciled​

The program now holds three persistence statements: the training rule 'LB persistence ≥ token timeouts' (Section 8.5); Nadine's deck 'source-IP or cookie persistence, timeout no less than 60 minutes' (Part III slide 52); and the official guide's two statements — the prerequisites page recommending 30-minute source-IP stickiness with a 'least connections' method and SSL-Passthrough, and the server-resources section specifying source-IP persistence for AMIS and cookie persistence for HDAP and SSP, with GSLB (if used) monitoring cluster status and pinning a user's session to one data center. Reconciled field guidance: (1) the per-service persistence types — source-IP for AMIS, cookie for the web portals — are the official design and should be the default; (2) the persistence timeout is governed by the training rule: set it ≥ the authconfig idle/lifetime values, which makes the 30- vs 60-minute figures two examples of the same principle at different token configurations, not a contradiction; (3) SSL handling: the guide recommends passthrough but explicitly permits SSL termination provided the LB re-encrypts before forwarding — and termination becomes necessary when cookie persistence is used (the LB must read the cookie). That last dependency — cookie persistence requires termination+re-encryption — is stated only in the official guide and belongs in every LB design conversation.

59.3 REFINEMENT — Health-Check URLs: The Authoritative Set​

Part III slide 52 gave AMIS /auth/mid; the official guide's prerequisites page gives /auth/statistics (expect body text authenticationStatistics), and its server-resources table gives both /auth/mid and a third option — /healthcheck/status.html (expect amis_server_online) — with equivalents for SSP and HDAP (/ssp/info/status / /hdap/info/status expecting startup_time, or their /healthcheck/status.html pages expecting ssp_server_online / hdap_server_online). The guide also supplies the monitor engineering parameters no other source has: HTTP monitor, 5-second polling interval, 3 consecutive responses to mark a server up or down, with pool-member ports 8443/8444/8445 behind 443 front-ends. Verdict: hand the customer's LB team the official guide's monitor spec verbatim; any of the three AMIS URLs works, and the /healthcheck/status.html family is the most uniform choice across all three services.

59.4 NEW AUTHORITY — AD Bind Account Requirements (Resolves Action D2-8)​

Day 2 referenced 'Justin's guide' for the AD bind permissions (Section 29.5, action D2-8). The official guide is that documentation, with screenshots. Requirements: the bind account must be a member of Domain Users; for password change/reset scenarios it must additionally hold delegated forest/domain access for both 'Change Password' and 'Reset Password', applied to Descendant User objects (the guide shows the exact Permission Entry dialogs); and the AM identity source must connect via LDAPS (the guide's Directory Connection screenshot shows ldaps:// on port 636 with a failover URL). The new critical gotcha, stated nowhere else in the program: AD password change/reset does not work via Global Catalog (port 3269) — an identity source bound to the GC will authenticate happily and fail only on password writes, a maddening partial-failure signature. Add to the R-14 mitigation: verify the identity-source URL targets a domain controller on 636, not the GC on 3269. D2-8 is hereby closed.

59.5 CONFIRMATION — PQS v2.0 Prompt Set Matches the Training​

The guide's Windows PQS walkthrough shows the current prompt set including the AM8 Authentication API Access Key (generated at Setup → System Settings → RSA SecurID Authentication API, communication port 5555) and the cloud block ending with the Private Key JWT file path and OAuth Client ID — exactly the OAuth/JWK model demonstrated live (Sections 5–6). The guide also confirms the multi-node pattern: run PQS only on the first internal instance; additional nodes are deployed by cloning it — a deployment rule the training implied but never stated as policy. One prompt-level footnote: the guide's Linux sample shows log level WARN as a default while training guidance says start at DEBUG (Section 6.3); the training guidance stands for new installs, with the Section 62 debug-cycle discipline governing when to come back down.

59.6 RECONCILIATION — Certificates: The JKS Procedure vs. the PEM Model​

The largest apparent conflict in the program: Day 2 training taught that modern PrimeKit uses PEM certificates (three files per app with the DH-parameters rule — Section 26), with JKS as the legacy customers' format; yet the official v2.0 guide's entire certificate-update procedure (12 pages) is JKS keystore-based — per-service keystores (amis_keystore.jks, hdap_keystore.jks, ssp_keystore.jks, plus the global truststore.jks), keytool-driven CSR/import workflow, and activation by pointing each setenv.sh keystore.file parameter at the new keystore. Reconciled verdict: both are true for different kit generations, and the kit in front of you decides — check server.xml/setenv.sh (Section 26.5): PEM-configured kits follow Section 26's procedure with the DH rule; JKS-configured kits follow the official guide's procedure. Facts from the JKS procedure worth having regardless: the default self-signed certs ship with CNs amis/ssp/hdap.company.com and keystore password 'password' (an obvious hardening item on any real deployment — changing filenames, passwords, or aliases requires matching setenv.sh updates); AMIS acts as client to AM, server to SSP/HDAP and integrators, and client/server when clustered (trust flows in both directions); SANs are added at key generation with the -ext switch and must be explicitly requested from the CA (with Microsoft CA web-form syntax provided: san:dns=name[&dns=name]); and self-signed defaults are acceptable for dev/test but must be replaced for production. A note for the Prime team (candidate addition to the P4/OQ list): the official guide predates or omits the PEM transition — worth flagging to Ramesh for the next guide revision.

59.7 CONFIRMATION — Developer Tooling Ships Today (Closes Action P4-1)​

Part IV raised whether the 2018-era test harnesses still ship (action P4-1). The official guide answers with screenshots: the Prime Authentication Harness (Base URL /auth; tabs for Authentication, Authorization, Validation, Statistics, New PIN/Next tokencode, Service Account — including RBA context, questions, SSO, and password flows) and the Prime REST/Service Harness (Base URL /am8; User Methods, Token Methods, direct-XML token update, create/update user, XML builder, group search — with compound operations, token assignment, and attribute search) are current PrimeKit Development Tools. The Advanced REST Client toolkit of Part IV slide 39 has been superseded by an official Postman collection of 50+ common AMIS REST APIs (Authentication and Administration folders spanning AM and CAS, including Auth Initialize/Verify and Version Info). P4-1 is closed: add both harnesses and the Postman collection to the consultant lab workflow (Section 19), and treat Part IV's ARC material as historical.

59.8 Minor Deltas Worth Recording​

  • AMIS internal HTTP 8080: the guide documents AMIS's HTTP port 8080 as localhost-only, servicing SSP and HDAP — the loopback path behind the ENS endpoint URL (Section 47.3) and the reason the port matrix says 'disable unless used' for external exposure.
  • Port matrix additions: SSH 22 Prime→AM (PQS dependency), Syslog 514/UDP to a syslog aggregator, RDP 3389 admin→Prime (Windows), CT-KIP web-tier path TCP 7022, LDAPS 636 to AD, and named RSA cloud egress IPs — extending Part III slide 10's matrix. ENS adds PostgreSQL 7050 (Chapter 61).
  • Four Template Builder templates: the tools package ships prime-kit-configuration templates for amis, ssp-internal, ssp-external, and hdap — the modern successors to Part IV slide 29's single template.
  • External SSP has no PQS: confirmed — external kit installation is scripts 1–3 plus a hand-edited config.sh, exactly as Part III slides 57–58 taught, with §60.7's must-match parameter list.
