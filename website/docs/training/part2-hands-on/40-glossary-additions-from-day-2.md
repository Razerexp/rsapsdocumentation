# 40. Glossary Additions from Day 2


The following terms supplement the Day 1 glossary (Section 20).

Term

Definition (as used in this training)

PEM

Base64-encoded, plain-text certificate format now standard in PrimeKit (cert, chain, and key files per application), replacing JKS

JKS

Java KeyStore — legacy certificate format still present at older customers; convert to PEM or replace in kind via server.xml

DH parameters

Diffie-Hellman key-exchange parameters block embedded in every application cert file; must be preserved (or regenerated) on every certificate replacement to avoid known TLS vulnerabilities

server.xml

Per-application Tomcat configuration (tomcat/tomcat-`<app>`/conf) whose SSL connector binds the cert, chain, and key files

config.sh / setenv.sh

Per-application runtime configuration pair holding ports, certificate/keystore references, encrypted bind credentials, and SMTP — the persisted form of every Quick Setup answer

ENC2

Prefix marking values in config.sh encrypted with the file's symmetric key (the 'MS debug war' line)

allowedUserProfileOperations

SSP properties line gating user-profile features; the generatePassword keyword enables the on-demand password button

A / a / N / S pattern

Password-generation composition syntax: uppercase, lowercase, numeric, symbol — minimums enforced, output shuffled

JSESSIONID

Standard web-session browser cookie used by SSP and HTAP (distinct from the AMS RSA session token); Redis-backed in containerized deployments

HAProxy

Load balancer container fronting MS/SSP/HTAP in the Docker topology

MailHog

Open-source mail-capture container used as a lab SMTP substitute for testing notification flows

Setup container

Run-once Docker container that executes Prime Quick Setup and exits

patches / backups folders

WAR-file patching mechanism: drop WARs in patches and restart to apply; automatic pre-patch copies in backups enable rename-and-restart rollback

VTS Builder

Administrator tool that white-labels the VTS client and emits the end-user MSI plus a server-side secret file

EAC (Emergency Access Code)

Short-lived code (~3 minutes) generated by placing a token in lost mode; VTS's per-request credential for non-persistent desktops

MTLS (two-way SSL)

Mutual certificate validation between VTS client and AMS server; client cert embedded in the MSI proves administrator provenance

softwareOnly

vtsconfig.xml flag restricting VTS to software tokens (hardware tokens refused)

pcapconfig.xml

PCAP configuration file (configs/AMS) governing which front-end authentication methods the portal offers

Central authentication section

am8config.xml block where SAML 2.0 / OIDC applications are integrated into PCAP — the IdP configuration surface

IdP-initiated / SP-initiated

SAML flow directions: launched from the PCAP portal vs. launched from the target application redirecting to PCAP

vCloud SecurID template

Full-stack lab template (~15–20 machines: AD\+DNS, AM primary/replica, IDR, Prime Windows/Linux, MFA agent hosts); 30-day renewal

common.css / style.css

The two files carrying all SSP/HTAP branding — colors, fonts, sizing, positioning; changes apply without restart

workflow/mail templates

Per-operation email template folders (HTML \+ properties pair each) under configs/MS; global folder holds the embedded logo

SOC 2 / SOC 3

Assurance reports available via engineering: SOC 3 public; SOC 2 typically under customer NDA
